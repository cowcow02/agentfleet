# AgentFleet — Session Log: 2026-04-08

## What Was Built

AgentFleet went from a concept discussion to a working, deployed platform in a single session.

### Starting Point
- Identified the gap: teams using AI coding agents locally have no centralized orchestration
- Discussed the microservice analogy (laptops as nodes), service discovery, agent registry
- Designed the entity model: Organization > Projects + Teammates + Agents (M:M assignments)
- Key insight: agent discovery model (agents defined at the edge, discovered by hub) vs runner model

### Infrastructure Built

#### Cloud Hub (Railway)
- **URL**: https://agentfleet-hub-production.up.railway.app
- **Repo**: https://github.com/cowcow02/agentfleet
- **Stack**: Node.js, WebSocket (ws), PostgreSQL, vanilla HTML/CSS/JS
- **CI/CD**: GitHub Actions (50 API tests + CLI smoke tests) → Railway auto-deploy on push to master
- **Database**: PostgreSQL on Railway with auto-migration

#### Hub Server (`hub/index.js`)
- Multi-tenant with team isolation
- Email/password auth (scrypt hashing) + session tokens (HMAC-signed, 7-day expiry)
- API tokens for daemon auth (backward compatible)
- Invite system with shareable codes
- Linear integration: webhook endpoint + API proxy for issue listing
- WebSocket server for daemon connections
- Stale connection cleanup (heartbeat-based, 15s interval)
- Write-through to PostgreSQL on every mutation
- Restore from DB on startup (survives redeploys)

#### Hub Pages (5 pages)
- **Landing** (`/`) — Sign in, create team, join via invite code
- **Dashboard** (`/dashboard`) — Stats, dispatch (Manual + From Linear tabs), fleet overview, live feed
- **Agents** (`/agents`) — Agent discovery registry with search/filter
- **Dispatches** (`/dispatches`) — Dispatch history with expandable timelines, duration tracking
- **Settings** (`/settings`) — Team members/tokens, Linear webhook config, invite generation

#### CLI (`cli/index.js`)
- `agentfleet login <token> --hub <url>` — authenticate, save config
- `agentfleet status` — show member, team, fleet info
- `agentfleet agents` — list local agent definitions
- `agentfleet setup` — local web UI (localhost:9901) for visual agent configuration
- `agentfleet start` — start daemon with saved config
- Installed globally via `npm link`

#### Daemon (`prototype/daemon/index.js`)
- Reads agents from `~/.agentfleet/agents.yaml` or `./agents.yaml`
- Reads hub/token from `~/.agentfleet/config.yaml` (set by CLI login)
- Connects to hub via WebSocket with token auth
- Registers agents on connect
- On dispatch: writes temp script, opens Ghostty (or any configured terminal), runs Claude
- Process lifecycle monitoring:
  - CPU-based idle detection (checks every 5s, reports idle after 20s of <2% CPU)
  - Duration tracking (reports on completion)
  - 30-second periodic status pings
- Status reported to hub: Running → Idle → Completed

#### Database (`hub/db.js`)
- PostgreSQL with connection pooling
- Auto-migration system (versioned, idempotent, runs on startup)
- Tables: teams, members, invites, dispatches, webhook_log, _migrations
- DATABASE_URL required — hub exits with clear error if missing or unreachable
- Write-through persistence on all mutations

#### Test Suite (`hub/test.js`)
- 50 API tests covering all endpoints
- Categories: health, auth (login/signup/session/tokens), teams, members, invites, dispatch, dispatches, metrics, webhooks, Linear settings/webhook, HTML pages, CORS
- Runs against real PostgreSQL (Docker locally, service container in CI)
- GitHub Actions: postgres:16-alpine service, Node 22

### Key Design Decisions

1. **Agent Discovery Model** — Agents defined locally (YAML), registered with hub on daemon connect. Hub discovers, doesn't define.
2. **Opaque Status Reporting** — Hub doesn't understand agent phases. Daemon pushes status strings. Hub timestamps and displays.
3. **Launcher Templates** — `{script}` variable interpolated into any terminal command (Ghostty, Terminal.app, iTerm2, tmux, headless). User controls how agents open.
4. **Interactive vs Headless** — Interactive mode opens full Claude TUI for human intervention. Headless mode pipes stream-json for autonomous operation.
5. **CPU Idle Detection** — Monitors child process CPU to distinguish "Claude is working" from "Claude is waiting for input."
6. **In-memory + Write-through** — All state in Maps for fast access, every mutation also writes to PostgreSQL for persistence.
7. **Team Isolation** — All data scoped by team. Agents, dispatches, members, settings — all filtered by team ID.

### Bugs Fixed During Session
- Foreign key constraint: team must save to DB before member (was fire-and-forget concurrent)
- SSL connection error: Railway internal Postgres doesn't use SSL
- Stale WebSocket connections: added heartbeat-based cleanup + ws.terminate()
- User info display: API returns `data.member.name` not `data.name`
- Landing page login: checking `data.member` not `data.name`
- Agent member column: `a.memberName` not `a.member_name`
- Settings team info: `data.team.name` not `data.team`
- CLI login/status display: same member/team field mapping

### Linear Integration
- Webhook configured at: `https://agentfleet-hub-production.up.railway.app/webhooks/linear/<teamId>`
- Trigger rules: when issue status matches configured status AND labels match
- Auto-dispatch: webhook → hub matches → dispatches to available agent → Ghostty opens
- Dashboard "From Linear" tab: fetches issues via Linear GraphQL API proxy
- Webhook log: every event recorded with action, reason, ticket info
