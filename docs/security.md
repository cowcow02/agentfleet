# Security

AgentFleet is designed with a security model that assumes zero trust between components and keeps sensitive data at the edge. This document describes transport security, authentication, authorization, event signing, and the overall trust boundary design.

## Security Principles

**The hub never touches code.** The hub is a coordination layer. It sends "work on ticket X" messages but never accesses git repositories, source code, credentials, or developer environments. All code execution happens on developer machines using their own local credentials and toolchains.

**Outbound-only connections.** Developer machines only make outbound connections to the hub. No inbound ports are opened on developer networks or machines. This eliminates an entire class of attack surface.

**Credentials stay local.** Git credentials, API tokens, environment variables, and SSH keys used by agents are local to the developer's machine. The hub does not see, store, or transmit them.

**Defense in depth.** Multiple layers of authentication, signing, and verification protect against different threat models.

## Transport Security

### Hub Connections

All connections between daemons and the hub use WebSocket over TLS (`wss://`).

```
Daemon  ------wss://------>  Hub
         outbound only
         TLS encrypted
```

Requirements:
- The hub must be served over HTTPS with a valid TLS certificate
- The daemon verifies the hub's TLS certificate on connection
- Self-signed certificates can be configured for private/internal deployments (not recommended for production)

### Webhook Connections

Tracker webhooks are received over HTTPS:

```
Jira/Linear/GitHub  ------HTTPS------>  Hub
                     webhook payload
                     signature header
```

The hub's webhook endpoint must be served over HTTPS. Webhook signatures are verified before processing (see Webhook Verification below).

### Local Connections

The daemon's local status endpoint listens only on `localhost`:

```
Agent process  ------HTTP------>  Daemon (localhost:7117)
               local only
               no network exposure
```

The local endpoint does not use TLS because it is not network-accessible. It binds exclusively to `127.0.0.1`.

## Authentication

### Three-Layer Authentication

AgentFleet uses three layers of authentication:

```
Layer 1: Team Token         -- identifies the organization
Layer 2: Machine Key        -- identifies a specific machine
Layer 3: HMAC Event Signing -- verifies individual messages
```

### Layer 1: Team Token

The team token (`af_xxxxxxxxxxxx`) is generated by the team lead and shared with developers. It identifies which organization a daemon belongs to.

Properties:
- Format: `af_` prefix followed by 32 random alphanumeric characters
- Scope: organization-wide (all machines in the organization use the same token)
- Rotation: can be rotated from the dashboard (all daemons must be reconfigured)
- Storage: stored in the daemon's config file (`~/.agentfleet/agents.yaml`)

The team token is sent in the WebSocket handshake as an HTTP header:

```
GET wss://hub.acme.dev/ws
Authorization: Bearer af_xxxxxxxxxxxx
X-Machine-Key: mk_xxxxxxxxxxxx
```

### Layer 2: Machine Key

The machine key (`mk_xxxxxxxxxxxx`) is generated during daemon initialization and uniquely identifies a specific developer machine.

Properties:
- Format: `mk_` prefix followed by 32 random alphanumeric characters
- Scope: machine-specific (each machine has its own key)
- Generation: created by the hub during `daemon init` enrollment
- Storage: stored in the daemon's config file

The machine key is sent alongside the team token in the WebSocket handshake. Both must be valid for the connection to be accepted.

This two-key system means:
- Knowing the team token alone is not sufficient to connect (need a registered machine key)
- A compromised machine key can be revoked individually without affecting other machines
- The hub can trace all activity to a specific machine

### Layer 3: HMAC Event Signing

Individual dispatch events are signed using HMAC-SHA256. This prevents message tampering on the WebSocket connection.

When the hub sends a dispatch:

```json
{
  "type": "dispatch",
  "dispatch_id": "dsp_abc123",
  "agent_name": "backend-agent",
  "ticket": { ... },
  "timestamp": "2026-04-08T09:00:00Z",
  "signature": "hmac_sha256_xxxxxxxxxxxx"
}
```

The signature is computed over the message body (excluding the signature field) using a shared secret derived from the machine key.

The daemon verifies the signature before acting on the dispatch. If verification fails, the dispatch is rejected and the event is logged as a security incident.

## Authorization

### Role-Based Access

The dashboard supports three roles:

| Role | Capabilities |
|------|-------------|
| **Admin** | Full access: organization settings, token management, project CRUD, routing rules, agent assignments, user management |
| **Lead** | Project management: create/edit projects, define routing rules, assign agents, view all dashboard data |
| **Viewer** | Read-only: view fleet status, pipeline, metrics, audit log |

### Machine Authorization

When a daemon connects, the hub verifies:

1. The team token is valid and active
2. The machine key is registered and not revoked
3. The machine key belongs to the organization identified by the team token

If any check fails, the connection is rejected.

### Agent Authorization

Agents are only eligible for dispatch if:

1. Their hosting machine is authenticated and connected
2. They are explicitly assigned to the project (via Assignment entity)
3. The assignment was created by a user with Lead or Admin role

Discovered agents that have not been assigned to any project cannot receive dispatches. This prevents unreviewed agents from being activated automatically.

## Webhook Verification

### Inbound Webhook Signatures

Each tracker integration has a webhook secret configured during setup. The tracker signs each webhook payload using HMAC-SHA256 with this secret, and includes the signature in an HTTP header.

| Tracker | Signature Header | Algorithm |
|---------|-----------------|-----------|
| Jira | `X-Hub-Signature` | HMAC-SHA256 |
| Linear | `Linear-Signature` | HMAC-SHA256 |
| GitHub | `X-Hub-Signature-256` | HMAC-SHA256 |

The hub verifies the signature before processing the webhook. If verification fails:
- The webhook is rejected with HTTP 401
- The event is logged in the audit log with the source IP
- The dashboard shows a security alert

### Replay Protection

Webhook payloads include a timestamp. The hub rejects payloads older than 5 minutes to prevent replay attacks. The clock tolerance is configurable for environments with clock skew.

## Trust Boundaries

```
+------------------------------------------------------+
|                    Developer Machine                  |
|                                                      |
|  +----------+    +--------+    +------------------+  |
|  | Agent    |--->| Daemon |    | Git credentials  |  |
|  | Process  |    |        |    | SSH keys         |  |
|  +----------+    +---+----+    | Environment vars |  |
|                      |         | Local toolchain  |  |
|                      |         +------------------+  |
+----------------------|-------------------------------+
                       |
              wss:// (outbound only)
              team token + machine key
              HMAC-signed events
                       |
+----------------------|-------------------------------+
|                      v                               |
|                    Hub                               |
|                                                      |
|  +-------------+  +-----------+  +---------------+   |
|  | Webhook     |  | Agent     |  | Dashboard     |   |
|  | receiver    |  | registry  |  | (TLS + auth)  |   |
|  +-------------+  +-----------+  +---------------+   |
|                                                      |
|  What the hub knows:                                 |
|  - Agent names, tags, capacity                       |
|  - Ticket metadata (titles, labels, priority)        |
|  - Status messages (opaque strings)                  |
|  - Timestamps and durations                          |
|                                                      |
|  What the hub does NOT know:                         |
|  - Source code                                       |
|  - Git credentials or SSH keys                       |
|  - Environment variables on developer machines       |
|  - Agent conversation content                        |
|  - File contents read/written by agents              |
|                                                      |
+------------------------------------------------------+
```

### Trust Boundary 1: Hub to Developer Machine

The hub sends dispatch messages containing ticket metadata (title, description, labels, priority). This is the same information visible in the project management tool. The hub does not send credentials, code, or environment data.

The daemon sends status messages containing opaque strings and completion data (exit code, PR URL). It does not send source code, diffs, or agent conversation logs.

### Trust Boundary 2: Daemon to Agent Process

The daemon spawns agent processes on the local machine. The agent process inherits the developer's environment (or the environment specified in the manifest). The daemon does not sandbox the agent -- it runs with the developer's permissions.

This is a deliberate design choice: agents need access to the local development environment (git, compilers, test frameworks, databases) to do meaningful work. Sandboxing would defeat the purpose of local-first execution.

### Trust Boundary 3: Hub to Tracker

The hub communicates with trackers using API credentials configured during project setup. These credentials are stored in the hub (encrypted at rest) and used for webhook verification, status transitions, and comment posting.

The hub should use credentials with minimal required permissions:
- **Jira:** Project-scoped bot account with "Transition issues" and "Add comments" permissions
- **Linear:** API key scoped to the relevant team
- **GitHub:** App installation scoped to the relevant repository with "Issues: write" permission

## Credential Storage

### Hub Credentials

The hub stores:
- Organization settings
- Tracker API credentials (encrypted at rest using AES-256-GCM)
- Team tokens (hashed, not stored in plaintext)
- Machine keys (hashed)
- Webhook secrets (encrypted at rest)

The encryption key is provided via environment variable (`AGENTFLEET_ENCRYPTION_KEY`) and should be managed through the deployment platform's secrets management.

### Daemon Credentials

The daemon stores in `~/.agentfleet/agents.yaml`:
- Team token
- Machine key
- Hub URL

File permissions should be restricted to the owning user (`chmod 600`).

Agent credentials (git tokens, API keys in `env` blocks) are part of the manifest and subject to the same file permissions. These are never transmitted to the hub.

## Threat Model

### Compromised Hub

If the hub is compromised, an attacker can:
- See ticket metadata (titles, descriptions, labels)
- See agent names, tags, and status messages
- Send fraudulent dispatch messages to connected daemons

Mitigations:
- HMAC signing on dispatch events (daemon rejects unsigned or incorrectly signed messages)
- Daemons only execute commands defined in their local manifest (a dispatch cannot inject arbitrary commands)
- The hub does not have access to source code, credentials, or development environments

### Compromised Daemon

If a daemon (developer machine) is compromised, an attacker has access to that developer's local environment, which is the same access the developer already has. AgentFleet does not increase the attack surface here.

Mitigations:
- Machine keys can be revoked from the dashboard
- The hub can detect anomalous behavior (unusual dispatch acknowledgment patterns)
- Other machines are not affected (no lateral movement via the hub)

### Compromised Team Token

If the team token is leaked, an attacker could attempt to register a rogue machine.

Mitigations:
- Registration requires hub approval (the team lead must assign agents to projects before they receive work)
- Token rotation from the dashboard
- Audit log shows all machine registrations

### Network Interception

If network traffic is intercepted:

Mitigations:
- All connections use TLS (hub-daemon, tracker-hub)
- HMAC signing on events provides integrity verification independent of transport encryption
- Webhook secrets prevent forged webhook payloads

## Audit Log

The hub maintains an append-only audit log of all security-relevant events:

| Event Type | Data Recorded |
|-----------|---------------|
| `machine.registered` | Machine name, IP, key hash, timestamp |
| `machine.connected` | Machine ID, IP, timestamp |
| `machine.disconnected` | Machine ID, reason, timestamp |
| `machine.revoked` | Machine ID, revoked by, timestamp |
| `agent.registered` | Agent qualified name, tags, capacity, timestamp |
| `agent.deregistered` | Agent qualified name, reason, timestamp |
| `dispatch.sent` | Dispatch ID, ticket ID, agent, timestamp |
| `dispatch.completed` | Dispatch ID, exit code, duration, timestamp |
| `dispatch.failed` | Dispatch ID, error, timestamp |
| `webhook.received` | Source tracker, project, signature valid, timestamp |
| `webhook.rejected` | Source IP, reason, timestamp |
| `auth.failed` | Source IP, reason, timestamp |
| `token.rotated` | Rotated by, timestamp |

The audit log is queryable from the dashboard and can be exported for external analysis.

## Operational Security Recommendations

**Rotate tokens periodically.** Team tokens should be rotated on a schedule (quarterly is a reasonable default) and immediately if a team member leaves.

**Review machine registrations.** When a new machine appears in the registry, the team lead should verify it belongs to a team member before assigning agents to projects.

**Use minimal tracker permissions.** Configure tracker API credentials with the minimum permissions required (issue transitions and comments, not full admin access).

**Monitor the audit log.** Set up alerts for: failed authentication attempts, webhook signature failures, unexpected machine registrations, and machines connecting from unusual IP ranges.

**Secure the hub deployment.** The hub should be deployed behind a reverse proxy with rate limiting, on a private network if possible, with access restricted to known IP ranges for the dashboard.
