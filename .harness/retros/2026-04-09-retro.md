## Findings

### Finding 1: Conversation files lack structure for friction tracking — APPLIED

- **Category:** harness gap
- **Phases:** all
- **Evidence:** AGE-6 conversation file has no `Harness Issues` section. Retro had to infer friction from absence of data.
- **Fix:** Added `## Harness Issues` template to the conversation file skeleton (pickup phase initializes it). Each phase skill now has an instruction to append friction entries with the documented format.

### Finding 2: Linear integration was read-only — APPLIED

- **Category:** harness gap
- **Phases:** pickup (and ship — already had partial sync from commit `1df6218`)
- **Evidence:** Pickup fetched the ticket but never moved it to "In Progress." Ship had recently been wired up to "In Review" + PR attachment. Round trip to "Done" was missing.
- **Fix:**
  - `harness-pickup`: added Linear status update to "In Progress" via `save_issue`
  - `harness-ship`: added Linear status update to "Done" after Railway deploy verification

### Finding 3: Ship phase ended at PR creation — APPLIED

- **Category:** harness gap
- **Phase:** ship
- **Evidence:** Ship phase took 1 min (push + `gh pr create`) and marked the ticket complete. No CI watch, no Railway verification. Broken CI or deploy failures only caught by humans.
- **Fix:** Extended `harness-ship`:
  - **Pre-review:** `gh pr checks --watch`, fix once on failure, escalate after 2 attempts
  - **Post-merge:** poll API + Web Railway healthcheck URLs (configurable via env, with sensible defaults), 5-min timeout
  - On healthcheck success: move Linear ticket to "Done"
  - Skip with note if Railway URLs not yet configured

### Finding 4: Verify phase had no runtime environment isolation — APPLIED

- **Category:** harness gap
- **Phase:** verify
- **Evidence:** Worktree isolation (added in `1df6218`) gave parallel agents separate filesystems, but `harness-verify` still used hardcoded API port `9900`, web port `3000`, and a single shared `agentfleet` database. Two parallel verifies would collide on ports and corrupt each other's DB state.
- **Fix:** Added per-task isolation scheme to `harness-verify`:
  - Slot = last 2 digits of task ID (zero-padded), e.g., AGE-6 → `06`
  - API port `99XX`, Web port `30XX`, DB name `agentfleet_age_XX`
  - Shared Postgres container reused; per-task DB created/dropped inside it
  - Dev servers spawned with `PORT`, `DATABASE_URL`, `WEB_URL`, `NEXT_PUBLIC_API_URL` env overrides
  - Browser navigation uses the dynamic web port
  - Verified that `apps/api` honors `PORT` env (`apps/api/src/env.ts`) and `drizzle.config.ts` honors `DATABASE_URL`, so no app code changes needed
  - Collision risk: AGE-6 and AGE-106 share slot `06` — rare in practice; surface explicitly if it happens

### Finding 5: Conversation files lost during ship — APPLIED (added mid-retro)

- **Category:** harness gap
- **Phase:** ship
- **Evidence:** AGE-11's conversation file was lost. Two compounding issues: (1) `harness-ship` step 1 said "stage all changed files" without explicitly naming the conversation file, so the agent missed it; (2) the `## Ship` section is recorded _after_ commit in step 9, so even with explicit staging the ship section never made the commit; (3) `git worktree remove` then ran without checking for unstaged/untracked files, destroying the work. `.harness/conversations/` is NOT gitignored — these files are meant to be committed.
- **Fix:**
  - `harness-ship` step 1: explicitly call out staging `.harness/conversations/<task-id>.md` with a "this file is part of the deliverable" note
  - New `harness-ship` step 10: after PR creation + CI watch + Railway healthcheck + Linear "Done" updates, do a follow-up commit/push capturing the ship-phase conversation file changes
  - `harness-ship` step 12 (was 11): add a hard safety check before `git worktree remove` — `git status --porcelain` must be empty, else surface to human and refuse to remove. **Never `--force`.**
